TradeShield Remote - Deployment Guide

Metadata

Category: Security and Trading Platform Administration Last Updated: 2024-11-29 Author: System Administrator Version: 1.0 Review Date: 2025-02-28 Status: Active

Purpose

This procedure covers the deployment of TradeShield Remote, a secure, high-availability remote administration system for critical trading platform infrastructure. The system provides encrypted VPN access via WireGuard and remote desktop capabilities via RustDesk.

When to Use: - Deploying new TradeShield Remote infrastructure - Setting up secure remote access to trading platforms - Implementing high-availability VPN gateway - Configuring remote administration cluster

Prerequisites

Required Access

  • [x] Hyper-V administrator privileges
  • [x] Domain administrative privileges
  • [x] Network configuration access
  • [x] SSH access to CHR instances

Required Tools

  • [x] Windows Server 2022 with Hyper-V role
  • [x] PowerShell 5.1+
  • [x] OpenSSH client
  • [x] WireGuard tools (for key generation)
  • [x] Network Load Balancing feature

Required Information

  • [x] Static IP range: 192.168.100.0/24
  • [x] External IP with ports 51820-51821 open
  • [x] MikroTik CHR VHD file (version 7.15+)
  • [x] Ubuntu Server 22.04 LTS ISO (for RustDesk servers)

Pre-Conditions

  • [x] Hyper-V host with 8GB+ RAM, 100GB+ storage
  • [x] Network switches configured (ExternalSwitch, InternalNetwork)
  • [x] External load balancer configured (HAProxy or equivalent)
  • [x] Network share for deployment files accessible

Procedure Steps

Step 0: Security Setup (Recommended Before Deployment)

Action: Set up security enhancements before or during deployment:

cd Scripts\06-Security-and-Trading-Platform-Administration

# Generate all keys first
.\Generate-TradeShield-Keys.ps1 -ClientCount 13

# Encrypt password storage
.\Setup-Encrypted-Password-Storage.ps1 -MigrationMode "Both"

Note: SSH key authentication setup should be done after CHR instances are deployed and accessible (see Step 3.5).

Step 1: Generate Secure Keys

Action: Generate all required cryptographic keys for the deployment:

cd Scripts\06-Security-and-Trading-Platform-Administration
.\Generate-TradeShield-Keys.ps1 -ClientCount 13

Expected Result: - Keys generated in C:\TradeShield\keys\ - CHR admin password created - WireGuard keys generated (primary, secondary, and 13 client keys) - File permissions set to restrict access

Verification:

Get-ChildItem C:\TradeShield\keys

Step 2: Create CHR Virtual Machines

Action: Create two MikroTik CHR virtual machines for high-availability VPN gateway:

.\New-TradeShield-CHR-VM.ps1 -CHRVersion "7.15"

Expected Result: - Two VMs created: TradeShield-CHR-01 and TradeShield-CHR-02 - VMs configured with 2 vCPUs, 2GB RAM each - External and internal network adapters configured - VMs set to auto-start

Verification:

Get-VM | Where-Object { $_.Name -like "TradeShield-CHR-*" }

Step 3: Configure CHR High Availability

Action: Configure both CHR instances with VRRP for high-availability:

.\Configure-TradeShield-CHR-HA.ps1 `
    -PrimaryIP "192.168.100.1" `
    -SecondaryIP "192.168.100.2" `
    -VirtualIP "192.168.100.254"

Expected Result: - Primary CHR configured with priority 254 - Secondary CHR configured with priority 100 - Virtual IP (192.168.100.254) configured via VRRP - WireGuard interfaces configured on both CHR instances - Secure admin user created, default admin disabled - SSH restricted to internal network (port 2222)

Verification:

# Test connectivity to Virtual IP
Test-Connection -ComputerName 192.168.100.254 -Count 3

Step 3.5: Set Up SSH Key Authentication (Recommended)

Action: Configure SSH key-based authentication for CHR instances:

.\Setup-SSH-Key-Authentication.ps1 -CHRIPs @("192.168.100.1", "192.168.100.2")

Expected Result: - SSH key pair generated (if not exists) - Public key distributed to CHR instances - Password authentication disabled - SSH key authentication working

Verification:

# Test SSH key authentication
ssh -i C:\TradeShield\keys\ssh\tradeshield-admin-key -p 2222 tradeshield-admin@192.168.100.1 "/system identity print"

Note: If plink is not available, follow manual instructions provided by the script.

Step 4: Deploy RustDesk Cluster

Action: Deploy RustDesk servers using the automated deployment script:

.\Deploy-RustDesk-Cluster.ps1 -NodeIPs @("192.168.100.10", "192.168.100.11")

Prerequisites: - SSH key authentication must be set up (see Step 1 in Security Setup) - RustDesk nodes must be accessible via SSH - User must have sudo privileges on RustDesk nodes

Expected Result: - Docker installed on both nodes - Docker Compose installed - RustDesk services (HBBS and HBBR) running in containers - Services accessible on ports 21115-21119 - Health checks configured

Verification:

# Test connectivity from Windows
Test-NetConnection -ComputerName 192.168.100.10 -Port 21115

# Or SSH to node and check
ssh -i C:\TradeShield\keys\ssh\tradeshield-admin-key tradeshield-admin@192.168.100.10 "docker ps --filter name=rustdesk"

Step 5: Configure Windows Network Load Balancing

Action: Configure Windows NLB for RustDesk cluster:

.\Configure-Windows-NLB.ps1 `
    -ClusterName "TradeShield-RustDesk" `
    -VirtualIP "192.168.100.20" `
    -Nodes @("192.168.100.10", "192.168.100.11")

Expected Result: - NLB cluster created - Both RustDesk nodes added to cluster - Port rules configured for ports 21115-21119 - Virtual IP (192.168.100.20) configured

Verification:

Get-NlbCluster
Get-NlbClusterNode
Get-NlbClusterPortRule

Step 6: Configure Firewall Rules

Action: Apply comprehensive firewall rules to CHR instances:

.\Configure-TradeShield-Firewall.ps1 `
    -CHRIPs @("192.168.100.1", "192.168.100.2") `
    -WGPorts @(51820, 51821)

Expected Result: - WireGuard rate limiting configured - SSH protection enabled - Trading platform access rules applied - Time-based restrictions for trading hours - Security event logging enabled

Verification:

# Test SSH access (should work from internal network)
ssh -p 2222 tradeshield-admin@192.168.100.1

Step 7: Test Failover Functionality

Action: Test high-availability failover mechanisms:

.\Test-TradeShield-Failover.ps1

Expected Result: - CHR failover test passes (VIP remains accessible) - RustDesk failover test passes (service remains accessible) - Test results logged to C:\TradeShield\logs\

Verification: Review test log file:

Get-Content C:\TradeShield\logs\failover-test-*.log | Select-Object -Last 20

Step 8: Set Up Monitoring Dashboard (Recommended)

Action: Deploy TradeShield Monitor dashboard:

# Navigate to monitor directory
cd TradeShield-Monitor

# Install dependencies
.\install.ps1

# Run in development mode (or build for production)
.\run.ps1 -Mode dev

Expected Result: - Go dependencies installed - React dependencies installed - Backend running on http://localhost:8080 - Frontend running on http://localhost:3000 - Dashboard accessible and displaying metrics

Verification: 1. Open browser to http://localhost:3000 2. Dashboard should show component status 3. Metrics should be updating

Alternative - Production Mode:

# Build for production
.\build.ps1

# Run in production mode
$env:ENV="production"
.\run.ps1 -Mode prod

Step 9: Set Up Automated Backups (Recommended)

Action: Configure automated backups:

# Set up automated backups
.\Backup-TradeShield-Automated.ps1 -RetentionDays 30 -EncryptBackups

Expected Result: - Backup scheduled task created (runs daily at 2:00 AM) - Backup directory structure created - Encryption enabled

Verification:

# Check scheduled tasks
Get-ScheduledTask | Where-Object { $_.TaskName -like "*TradeShield*" }

# Check backup directory
Get-ChildItem C:\TradeShield\backups

Verification

How to Verify Success

  1. CHR High Availability:
  2. Virtual IP (192.168.100.254) is reachable
  3. Primary CHR has priority 254
  4. Secondary CHR has priority 100

  5. VRRP advertisements are working

  6. RustDesk Services:

  7. Both RustDesk servers are running
  8. Services accessible via Virtual IP (192.168.100.20)

  9. Load balancer distributes traffic correctly

  10. Firewall Configuration:

  11. WireGuard connections are rate-limited
  12. SSH access restricted to internal network

  13. Trading platform ports are accessible from WireGuard network

  14. Failover Testing:

  15. CHR failover completes within 30 seconds
  16. RustDesk failover maintains service availability
  17. All tests pass

Success Criteria

  • [x] Both CHR instances configured and running
  • [x] Virtual IP accessible and failover working
  • [x] RustDesk cluster operational
  • [x] Load balancer configured and distributing traffic
  • [x] Firewall rules applied and logging enabled
  • [x] Failover tests pass

Troubleshooting

Common Issues

Issue 1: CHR VM fails to start

Symptoms: - VM created but won't start - Boot errors in VM console

Solution: 1. Verify CHR VHD file exists and is valid 2. Check Hyper-V integration services 3. Verify network switches exist 4. Check VM generation (must be Generation 2)

Prevention: - Use official MikroTik CHR VHD files - Verify file integrity before deployment

Issue 2: CHR configuration fails via SSH

Symptoms: - SSH connection timeout - Configuration commands fail

Solution: 1. Verify CHR is fully booted (wait 2-3 minutes after VM start) 2. Check network connectivity: Test-Connection -ComputerName 192.168.100.1 3. Verify default admin credentials (admin/admin) 4. Check SSH service is enabled on CHR

Prevention: - Wait for full CHR boot before configuration - Test connectivity before running configuration script

Issue 3: VRRP not working

Symptoms: - Virtual IP not accessible - Both CHR instances show as master

Solution: 1. Verify VRRP interface configuration on both CHRs 2. Check VRRP group ID matches (should be 50) 3. Verify priorities are different (254 vs 100) 4. Check network connectivity between CHR instances

Prevention: - Use consistent VRRP configuration - Verify network switches allow multicast traffic

Issue 4: RustDesk services not accessible

Symptoms: - Cannot connect to RustDesk via Virtual IP - Services not responding

Solution: 1. Verify Docker containers are running: docker ps 2. Check port bindings: docker port <container> 3. Verify firewall rules allow traffic 4. Test direct connection to node IPs

Prevention: - Monitor container health - Set up container restart policies - Configure health checks

Issue 5: NLB cluster creation fails

Symptoms: - NLB module not found - Cluster creation errors

Solution: 1. Install NLB feature: Install-WindowsFeature NLB 2. Verify network adapter name matches 3. Check for existing NLB clusters 4. Ensure nodes are on same network segment

Prevention: - Install NLB feature before deployment - Use consistent network adapter names

Notes

WARNING: This deployment affects critical trading infrastructure. Ensure proper testing in non-production environments before production deployment.

SECURITY WARNING: Before production deployment, review and address security concerns documented in TradeShield Remote - Security Review. Critical security issues include: - Plain text password storage (use encrypted storage) - SSH password authentication (implement key-based auth) - Missing audit logging (configure centralized logging) - No automated security monitoring (implement IDS/IPS)

AVAILABILITY WARNING: Current deployment has single Hyper-V host creating a single point of failure. Consider implementing Hyper-V cluster for production.

TIP: Use the failover test script regularly to verify high-availability is working correctly.

NOTE: Linux-based scripts for RustDesk deployment are provided separately and should be run on Ubuntu servers.
